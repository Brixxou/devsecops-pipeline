GitHub Actions — Le socle

name: Pipeline DevSecOps

on:
  push:
    branches: ["main"]
  pull_request:
    branches: ["main"]

env:
  IMAGE_NAME: devsecops-web
  IMAGE_TAG: latest

jobs:
  check-secrets:    # Etape 1 - parallele
  check-quality:    # Etape 2 - parallele
  sast-analysis:    # Etape 3 (needs: 1, 2)
  sca-analysis:     # Etape 4 (needs: 1, 2)
  infra-analysis:   # Etape 5 (needs: 3, 4)
  dast-analysis:    # Etape 6 (needs: 5)
  sign-image:       # Etape 7 (needs: 6, main only)
Principe : Chaque etape est un job isole. Les dependances (needs) definissent l'ordre. Si une etape critique echoue, la pipeline s'arrete.

Etape 1

Secrets — TruffleHog

Pourquoi ?

  • Un secret pousse dans Git reste dans l'historique pour toujours
  • Cles API, mots de passe BDD, tokens SSH, certificats...
  • Un seul leak = compromission totale du service

Comment ?

  • Scan de tout l'historique Git (fetch-depth: 0)
  • Detection par entropie + patterns connus
  • Verification active aupres des providers (AWS, GitHub...)
5
Etape 1

Secrets — Code Pipeline

check-secrets:
  runs-on: ubuntu-latest
  steps:
    - name: Recuperation du code
      uses: actions/checkout@v4
      with:
        fetch-depth: 0  # Historique complet

    - name: Scan des Secrets (TruffleHog)
      uses: trufflesecurity/trufflehog@main
      with:
        extra_args: --only-verified
--only-verified : Ne bloque la pipeline que sur les secrets confirmes comme valides via appel API reel. Reduit les faux positifs.

Etape 2

Linter — PHP_CodeSniffer

Pourquoi ?

  • Code propre = moins de bugs = moins de vulnerabilites
  • Standard PSR-12 pour PHP
  • Lisibilite et maintenabilite du code

Ce qu'il detecte

  • Indentation incorrecte (4 espaces attendus)
  • Espaces manquants : if( → if (
  • Accolades mal positionnees
  • Espaces en fin de ligne
check-quality:
  runs-on: ubuntu-latest
  steps:
    - uses: actions/checkout@v4
    - name: Linter PHP (PSR-12)
      run: |
        docker run --rm -v "$(pwd)/src":/data cytopia/phpcs --standard=PSR12 --ignore=vendor/* /data

Etape 3

SAST — Code Pipeline

sast-analysis:
  needs: [check-secrets, check-quality]
  runs-on: ubuntu-latest
  container:
    image: returntocorp/semgrep
  steps:
    - uses: actions/checkout@v4
    - name: Scan SAST (Semgrep)
      run: semgrep scan --config=p/php --error --verbose
--error : Retourne un code d'erreur si des vulnerabilites sont trouvees → bloque la pipeline.
--config=p/php : Regles communautaires specifiques a PHP.

Etape 4

SCA — Trivy FS

Supply Chain Security

  • Analyse des dependances (composer.lock)
  • Compare avec les bases de CVE connues
  • Identifie les composants obsoletes ou non maintenus

Resultat sur vulnweb

  • phpmailer 5.2.18 → CVE critiques
  • Correction : mise a jour vers ^6.0
  • Supprimer la section config.audit
sca-analysis:
  needs: [check-secrets, check-quality]
  steps:
    - uses: actions/checkout@v4
    - name: Installation des dependances
      run: docker run --rm -v "$PWD/src":/app composer install --ignore-platform-reqs
    - name: Scan SCA (Trivy FS)
      uses: aquasecurity/trivy-action@master
      with:
        scan-type: 'fs'
        scan-ref: './src'
        severity: 'CRITICAL,HIGH'

Etape 5

Infrastructure — Trivy Image

Pourquoi scanner l'image ?

  • php:7.2-apache = Debian 10 obsolete
  • 1600+ vulnerabilites dont 86 critiques
  • Un RCE + failles noyau = escalade triviale

Remediation

  • Upgrade vers php:8.2-apache
  • Resultat : 0 CRITICAL
  • Encore mieux : image Alpine FPM
infra-analysis:
  needs: [sast-analysis, sca-analysis]
  steps:
    - uses: actions/checkout@v4
    - name: Build de l'image Docker
      run: docker build -t $IMAGE_NAME:$IMAGE_TAG ./src
    - name: Scan Infrastructure (Trivy Image)
      uses: aquasecurity/trivy-action@master
      with:
        scan-type: 'image'
        image-ref: '$IMAGE_NAME:$IMAGE_TAG'
        severity: 'CRITICAL,HIGH'

Etape 6

DAST — ZAP Baseline

Analyse Boite Noire

  • Teste l'application en cours d'execution
  • Simule le comportement d'un attaquant reel
  • Intervient sur un environnement de staging

Vulnerabilites trouvees

  • XSS reflechis confirmes
  • Injection SQL confirmee
  • Headers de securite manquants
  • CSRF potentiel
dast-analysis:
  needs: [infra-analysis]
  steps:
    - uses: actions/checkout@v4
    - name: Demarrage de la stack
      run: |
        docker compose up -d --build
        sleep 15
    - name: Scan DAST (ZAP Baseline)
      uses: zaproxy/action-baseline@v0.14.0
      with:
        target: 'http://localhost:8080'
    - name: Arret de la stack
      if: always()
      run: docker compose down

Etape 7

Signature d'image — Cosign

Le probleme

  • Comment garantir que l'image en prod est celle validee par la pipeline ?
  • Risque d'alteration entre build et deploiement
  • Supply chain attack sur le registry

Sigstore / Cosign

  • Signature cryptographique keyless
  • Liee a l'identite OIDC du workflow GitHub
  • Verifiable par n'importe qui
sign-image:
  needs: [dast-analysis]
  if: github.event_name == 'push' && github.ref == 'refs/heads/main'
  permissions:
    id-token: write  # Necessaire pour keyless signing
  steps:
    - uses: sigstore/cosign-installer@v3
    - run: cosign sign --yes $USER/$IMAGE:$TAG
    - run: cosign verify $USER/$IMAGE:$TAG

Admission Controller

Kyverno — Gardien du cluster

Role

  • Bloque le deploiement si l'image n'est pas signee
  • Refuse les conteneurs root
  • Exige des limites CPU/RAM
  • Applique les politiques au niveau cluster

Exemple de politique

apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: verify-image
spec:
  rules:
    - name: check-signature
      verifyImages:
        - attestors:
            - keyless:
                issuer: "https://token.actions.githubusercontent.com"

Runtime Security

Falco — Detection d'intrusion

Principe

  • Surveille les appels systeme en temps reel
  • Detecte les comportements suspects meme si le code etait propre
  • Alertes immediates (Slack, PagerDuty...)

Exemples de detection

EvenementAlerte
Shell dans un conteneur webCRITICAL
Lecture de /etc/shadowCRITICAL
Connexion sortante inattendueHIGH
Modification de binairesHIGH
Complementaire au DAST : Le DAST teste avant le deploiement. Falco protege apres, en production.

Scan continu

OpenVAS / Nessus — Scan recurrent

Le probleme

  • Une nouvelle CVE sort apres le deploiement
  • Pas de redeploiement = la pipeline ne la verra jamais
  • L'infrastructure vieillit en silence

La solution

  • Scans planifies (quotidiens / hebdomadaires)
  • Inventaire complet des actifs
  • Alertes sur nouvelles vulnerabilites
  • OpenVAS Nessus Qualys
